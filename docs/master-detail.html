AISE FS26 · Personenverwaltung · 4 Ansätze
Ansatz 1
HTML-Form mit vollem Server-Roundtrip: Das Formular sendet einen POST-Request an den Server. Der Server verarbeitet die Daten und sendet die gesamte HTML-Seite neu zurück. Einfachste Variante — kein JavaScript nötig, aber jedes Speichern lädt die ganze Seite neu.
Hier simuliert: Der «Server» ist ein <form action=""> mit method="get", das die URL-Parameter ausliest und die Tabelle neu rendert.

Ansatz 2
Ajax Partial Update: Beim Klick auf eine Tabellenzeile sendet der Browser einen XMLHttpRequest (oder fetch) an den Server. Der Server gibt nur das Formular-HTML zurück — nicht die ganze Seite. Der Browser ersetzt den #master-region-Container mit dem Server-HTML. Klassisches AJAX-Muster (ca. 2005–2015).
Hier simuliert: Der «Server» gibt ein vorgefertigtes HTML-Fragment zurück.

Ansatz 3
fetch() mit JSON-API: Beim Zeilen-Klick ruft JavaScript fetch('/api/persons/{id}') auf. Der Server antwortet mit JSON. JavaScript liest das JSON aus und befüllt die Formularfelder direkt im DOM — kein HTML vom Server, nur Daten. Moderner Standardansatz für Web-Apps ohne Framework.
Hier simuliert: fetch() wird gegen eine lokale Mock-Funktion ersetzt, die nach 300 ms ein Promise mit JSON auflöst.

Ansatz 4
JavaScript State Management (kein Server-Roundtrip): Alle Daten leben im Browser-Speicher (persons[] Array). Auswahl, Editieren, Hinzufügen und Löschen passieren vollständig im Client. Die UI wird reaktiv neu gerendert, wenn sich der State ändert — Grundprinzip von React, Vue, Svelte. Kein Netzwerkaufruf nötig.
